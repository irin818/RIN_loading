# Runtime Dataflow Trace

Status: v2 Python-only dataflow analyzer.

## Purpose

Runtime Trace records safe, structured dataflow metadata for each real chat turn
as it moves through the Python backend. It exists for debugging, development
inspection, model behavior analysis, memory/context verification, thinking leak
analysis, and pipeline correctness checks.

The trace is generated by backend instrumentation inside the actual conversation
runtime. The UI does not fabricate flow steps.

## Trace v2 Stage Model

Each stage records the same high-level sections:

```python
{
    "name": "context_assembly",
    "displayName": "Context Assembly",
    "status": "ok|skipped|warning|error",
    "durationMs": 0,
    "summary": "2 messages / 96 chars",
    "input": {},
    "operation": {},
    "output": {},
    "decision": {},
    "privacy": {},
    "warnings": [],
    "errors": [],
}
```

This lets the owner inspect what entered a stage, what the backend did, what came
out, why the result happened, and what was hidden for privacy.

## Recorded Stages

The current trace records these stages in order:

- `input_received`
- `owner_message_persisted`
- `profile_loading`
- `recent_history_selection`
- `memory_v2_retrieval`
- `context_assembly`
- `model_request`
- `raw_model_response`
- `sanitization_final_answer`
- `rin_reply_persisted`
- `memory_update`
- `response_returned`

If a stage is not part of the active runtime yet, it is recorded as `skipped`
with an explicit reason. For example, `memory_v2_retrieval` is currently
recorded as skipped because the runtime writes Memory V2 traces after a turn but
does not yet retrieve Memory V2 traces for prompt assembly.

## Real Data Shown

The analyzer shows safe real data from the backend:

- input length, preview, hash, role, target conversation, and normalization
  status;
- owner/RIN database write status, message ids, content lengths, and hashes;
- profile report status, files loaded, summary keys, and context character count;
- recent-history selection policy, available/selected counts, excluded counts,
  timestamps, selected message metadata, previews, and hashes;
- Memory V2 retrieval status and explicit skip reason;
- Context V2 component order, component table, included flags, character counts,
  drop/dedup counts, and budget;
- model adapter/provider/model/options and request outline by role and length;
- raw model response duration, length, hash, safe preview, and thinking flags;
- sanitizer rules, raw/final lengths, removed character count, rejection status,
  and final preview;
- Memory V2 update attempt, created trace id, trace count delta, signal count
  delta, and full-text storage status;
- UI/API response status, total duration, final answer length, conversation id,
  and message id.

## Storage

Runtime traces are stored in process memory only:

- latest recent traces are retained for the running server session;
- traces are lost on server restart;
- no database schema changes are introduced;
- no trace data is written into `.rin-data/` beyond the existing runtime writes.

## Privacy Mode

Runtime Trace currently supports safe mode only.

Safe mode includes:

- ids;
- counts;
- timestamps;
- lengths;
- hashes;
- short previews;
- stage status;
- reason codes and safe metadata.

Safe mode does not include:

- full owner message text;
- full raw prompt;
- full memory text;
- full profile text;
- full raw model output;
- hidden reasoning.

Debug reveal mode is intentionally not implemented yet. A future version may add
a local-only gate such as `RIN_TRACE_SHOW_PRIVATE=allow`, but that should remain
explicit and must never expose secrets.

## API

Read-only endpoints:

```text
GET /api/diagnostics/runtime-trace
GET /api/diagnostics/runtime-trace/latest
GET /api/diagnostics/runtime-trace/{turn_id}
```

Responses are safe by default and include:

- `privacyMode: "safe"`;
- `fullTextIncluded: false`;
- `rawPromptIncluded: false`;
- `rawModelOutputIncluded: false`;
- trace status and safe stage metadata.

The endpoints do not make model calls and do not write to the database.

## UI

Open the Control Console and select:

```text
Runtime Trace
```

The page shows:

- compact latest-turn summary with short ids;
- compact vertical pipeline timeline with status and per-stage summary;
- end-to-end diagnostics for input, recent history, memory injection, request
  messages, raw output, removed characters, and sanitized storage;
- CSS-only bars for raw-to-final sanitizer shrinkage, context-to-request
  composition, and stage duration;
- safe trace JSON behind a collapsed disclosure.

Click any stage in the timeline to open a free-floating inspector window.
Multiple stage windows can be open at once, so the owner can compare, for
example, Context Assembly and Model Request side by side. Each window is appended
to a global fixed-position layer near the end of the page, can be moved by its
title bar, focused by clicking it, and closed independently. The Runtime Trace
toolbar can close all inspector windows or reset their layout.
`Esc` closes the topmost inspector. `Shift+Esc` closes all open inspectors.

This global layer is deliberate: earlier inspector windows were rendered inside
the scrollable glass console panel, where layout and clipping ancestors could
make dragging feel constrained or cause windows to jump back toward their
original panel position.

Inspector close controls stop pointer propagation before drag handling starts,
and drag listeners are removed on pointerup or pointercancel. Safe JSON inside
an inspector is rendered lazily only when its disclosure is opened, reducing UI
freeze risk when many stage windows are open.

Inspector windows are stage-specific rather than generic metadata dumps. Each
window shows a short purpose line, curated primary fields, targeted tables or
visuals when useful, diagnostics, privacy status, and collapsed safe JSON. For
example:

- Recent History shows selected message metadata as a compact table.
- Context Assembly shows the context component table.
- Model Request shows request options and request outline.
- Sanitizer shows raw-to-final length reduction.
- Memory Retrieval prominently shows when retrieval is skipped because Memory V2
  is not wired into prompt assembly yet.

All inspector content uses only the safe backend trace data already rendered into
the page.

If no chat turn has run in the current server session, the page shows an empty
state and asks the owner to send a message in Chat / Test.

## Thinking Leak Observability

Trace naming is intentionally conservative:

- `Provider Response` records safe raw provider metadata when the adapter can
  expose it, such as raw length, raw hash, thinking flags, and whether the
  adapter sanitized content. It does not include full raw output.
- `Sanitizer` records the runtime's final validation of the adapter content and
  the final answer that may be stored.
- `Store Reply` records only the sanitized answer storage result.

For mock adapters that do not expose raw provider metadata, Runtime Trace marks
`providerRawMetadataAvailable` as `false` instead of pretending the adapter
content is true raw provider output.

The trace records safe sanitizer fields:

- `thinkingTagDetected`;
- `thinkingLikePrefixDetected`;
- `sanitizerApplied`;
- `thinkingTagRemoved`;
- `thinkingLikePrefixRemoved`;
- `removedCharacterCount`;
- `rawLength`;
- `finalLength`;
- `finalAnswerPreview`;
- `storedSanitizedAnswer`;
- `storedRawThinking`.
- `extractedFinalAnswer`;
- `rejectionReason`.

This makes thinking leaks observable without storing hidden reasoning in memory
or exposing raw model output in the UI.

The Model Request stage also records whether the current owner message is present
and last in the request. A warning is added if that invariant ever fails.

Recent history is injected into the model request as bounded short-term context:
up to six prior messages, with each message capped and total recent-history
context capped. Diagnostics expose only counts, truncation status, hashes, and
short previews by default.

## Current Limitations

- Runtime traces are in-memory only.
- Safe mode is the only implemented display mode.
- The active runtime does not retrieve Memory V2 traces for prompt assembly yet;
  the trace records that stage as skipped instead of inventing retrieved memory.
- Profile loading is recorded through the existing profile report summary; full
  profile text is not exposed.
- The sanitizer still reflects the current implementation. It does not claim to
  catch all possible Qwen3 reasoning leak patterns.
