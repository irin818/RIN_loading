# Runtime Dataflow Trace

Status: first Python-only implementation.

## Purpose

Runtime Trace records safe metadata for each real chat turn as it moves through
the Python backend. It exists for debugging, development inspection, model
behavior analysis, memory/context verification, thinking leak analysis, and
pipeline correctness checks.

The trace is generated by backend instrumentation inside the actual conversation
runtime. The UI does not fabricate flow steps.

## Recorded Stages

The current trace records these stages in order:

- `input_received`
- `owner_message_persisted`
- `profile_loading`
- `recent_history_selection`
- `memory_v2_retrieval`
- `context_assembly`
- `model_request`
- `raw_model_response`
- `sanitization_final_answer`
- `rin_reply_persisted`
- `memory_update`
- `response_returned`

If a stage is not part of the active runtime yet, it is recorded as `skipped`
with an explicit reason. For example, `memory_v2_retrieval` is currently
recorded as skipped because the runtime writes Memory V2 traces after a turn but
does not yet retrieve Memory V2 traces for prompt assembly.

## Storage

Runtime traces are stored in process memory only:

- latest recent traces are retained for the running server session;
- traces are lost on server restart;
- no database schema changes are introduced;
- no trace data is written into `.rin-data/` beyond the existing runtime writes.

## Privacy Mode

The first version supports safe mode only.

Safe mode includes:

- ids;
- counts;
- timestamps;
- lengths;
- hashes;
- short previews;
- stage status;
- reason codes and safe metadata.

Safe mode does not include:

- full owner message text;
- full raw prompt;
- full memory text;
- full profile text;
- full raw model output;
- hidden reasoning.

Debug reveal mode is intentionally not implemented yet. A future version may add
a local-only gate such as `RIN_TRACE_SHOW_PRIVATE=allow`, but that should remain
explicit and must never expose secrets.

## API

Read-only endpoints:

```text
GET /api/diagnostics/runtime-trace
GET /api/diagnostics/runtime-trace/latest
GET /api/diagnostics/runtime-trace/{turn_id}
```

Responses are safe by default and include:

- `privacyMode: "safe"`;
- `fullTextIncluded: false`;
- `rawPromptIncluded: false`;
- `rawModelOutputIncluded: false`;
- trace status and safe stage metadata.

The endpoints do not make model calls and do not write to the database.

## UI

Open the Control Console and select:

```text
Runtime Trace
```

The page shows:

- selected/latest turn summary;
- vertical pipeline timeline;
- stage detail panels;
- request/response metrics through stage metadata;
- sanitizer status;
- memory/context summaries;
- safe trace JSON behind a collapsed disclosure.

If no chat turn has run in the current server session, the page shows an empty
state and asks the owner to send a message in Chat / Test.

## Thinking Leak Observability

The trace records safe sanitizer fields:

- `thinkingTagDetected`;
- `thinkingLikePrefixDetected`;
- `sanitizerApplied`;
- `thinkingRemoved`;
- `removedCharacterCount`;
- `rawContentLength`;
- `finalAnswerLength`;
- `storedSanitizedAnswer`;
- `storedRawThinking`.

This makes thinking leaks observable without storing hidden reasoning in memory
or exposing raw model output in the UI.

## Current Limitations

- Runtime traces are in-memory only.
- Safe mode is the only implemented display mode.
- The active runtime does not retrieve Memory V2 traces for prompt assembly yet;
  the trace records that stage as skipped instead of inventing retrieved memory.
- Profile loading is recorded through the existing profile report summary; full
  profile text is not exposed.
- The sanitizer still reflects the current implementation. It does not claim to
  catch all possible Qwen3 reasoning leak patterns.
